# PRD: 局卡 event-card（核心通用组件 / cornerstone）

> 见 [README](./README.md) 的成局逻辑与跨页契约。本轮只定产品功能范围，不含实现。新功能改本文件。

## 目的
JU 的核心可分享件 = **多状态数码票根**，融合 *分享入口(二维码)* + *个人化票根* + *成局进度*。
模块化、角色感知；用于 **dashboard 顶 / 管理台 / 局详情 / 分享**。是 §1/§8"数码票根"的具体化。

## 当前功能（已建）
无独立局卡组件。现有"局卡"只是发现页/仪表盘的静态信息卡（封面/标题/主办/时间/城市），无状态、无二维码、无成局信息。

## MVP 需要的功能
- **两态 + 过渡动画**（第三态 = 后续）：
  - **态1 正面·艺术视觉**：**时间 + 地点**以艺术字体嵌入卡片艺术风格 **+ 右下角小二维码**（扫码进局详情）。**不含局名/主办**（局名靠分类设计的视觉传达）。即分享/保存的那一面。
  - **态2 个人化·进度**：**你的状态**（留位中 / 等确认 / 已锁定席位）+ **成局进度**（已 N 人 + 缺 X 人，人数看板非名单）+ 已成局标识。视觉 TBD。
- **交互（host/访客同构）**：态1 → 点 → 态2 → **再点一次展开表单**：访客 = 编辑个人 RSVP 信息；host = 管理表单。
- **流程**：
  - host 发布后 → 第一时间呈现局卡 + **保存按钮**（长按保存正面含二维码图、用于转发）。
  - host 再登录 → 局卡从态1起，两态过渡。
  - 访客扫码 → 局详情页（局卡顶 + 留位表单下）→ 填完保存 → 整页变个人化局卡 → 点开编辑（见 [event-page](./event-page.md)）。
- **host 可同时是参加者**（也能留位）。
- **建局即选局卡**：建局时选分类 → 平台**自动生成多款局卡**供 host 选（**不传图**，审美兜底）；见 [event-create](./event-create.md)。

## 缓做 / 不做
- 态3 背面大二维码；单独"局卡页"；Apple/Google 钱包化；host 传图包装进局卡；分类 art 模板（Step 10B）。
- 态2 视觉设计（仅定信息，视觉后改，Step 10B）。

## 实现逻辑（Step-10A 设计）
- **数据**：大部分复用 R4 的 `get_event_by_slug`（is_locked/unlocked/capacity_remaining/host_wechat_id）+ token + 轮询；`get_event_by_slug` 补返回 `category`/`card_variant`。
- **二维码** = 编码 `/[slug]` 绝对链接、客户端 QR 库生成、手机相机扫码 → `/[slug]`（无站内扫码器）。
- **态1 正面** = 服务端 PNG（`next/og` `ImageResponse`，嵌二维码；art 布局=Step 10B）；**兼作 OG 分享图**（替代原封面 og:image；现 `og.ts` 仅产元数据、需新增图片路由）。
- **态2** = 交互 DOM（实时缺X人/你的状态/展开表单）；切换=淡入/翻转。
- **角色**（host/guest→展开管理 vs 编辑）由页面上下文 + auth 决定；展开复用 R4 的 submit_rsvp/lock_event/get_event_guest_contacts。
- **新件**：局卡共享组件 + `next/og` 图片路由 + QR 生成库。
